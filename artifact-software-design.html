Narrative

Enhancement narrative

The full Milestone Two narrative for the software design and engineering enhancement.

Describe the artifact

The artifact is an inventory management application I originally built for CS 360, Mobile Architecture and Programming. The original is an Android app written in Java with three screens: a login screen, an inventory list, and an add-item form. It uses a Material 3 interface and the navigation between screens works, but the back end was never finished. The login screen accepts any username and password as long as both fields have text, with a comment in the source admitting the credential check was left for a later project. There is no database, so the inventory screen has nothing to display and the add-item form shows a confirmation message without saving anything. An SMS alerts toggle is half-wired, with the Android permission flow in place but no code that actually sends a message. For Milestone Two I rebuilt this artifact as a working web application.

Justify the inclusion of the artifact and the improvements made

I selected this artifact because its weaknesses are exactly the kind of thing a software design enhancement should fix, and because rebuilding it lets me show full-stack work rather than a single layer. The enhanced version is a React single-page frontend with a Node.js and Express backend and an embedded SQLite database. The three Android screens map to three React routes. The fake login is now real authentication: passwords are hashed with bcrypt, and a successful login returns a JSON Web Token that the client sends on every protected request. The empty inventory screen is now a searchable table backed by a real Express API, and the add-item form is now a real endpoint that validates its input and writes to the database. I added role-based access control with two roles, admin and staff, enforced by middleware on the server: any authenticated user, admin or staff, can read the inventory and add new items, while updating and deleting an item are restricted to an admin. Every database query uses parameterized statements, which prevents SQL injection, and every endpoint that accepts input validates it with the Zod schema library before touching the database. The unused Android permissions are simply gone, because the web version has no equivalent attack surface. A seed script, seed.js, populates the database with two demo accounts and fifteen realistic inventory items so the application can be run and reviewed immediately.

Service-layer extraction and the thin-route pattern

The most important structural change is one I made after Dr. Bolton's Milestone Two feedback. In the version I first submitted, the Express route handlers did everything: they parsed the request, ran the business rules, called the SQLite database directly through db.prepare, and shaped the response, all in one function. I refactored that into a dedicated service layer so the routes are now thin. A route now parses and validates the request, calls a service function, and shapes the response, and nothing more. The business logic lives in three new modules. authService.js owns account creation and login, including bcrypt hashing, JWT signing, and the rule that the first user to register becomes the admin while every later account defaults to staff. Its public surface is two functions, registerUser and loginUser. inventoryService.js owns the inventory operations, listItems, createItem, updateItem, and deleteItem, along with the SKU-uniqueness rule and the existence checks. serviceError.js defines a small ServiceError class that carries an HTTP status code but knows nothing about Express, so a service can report a duplicate SKU or a taken username without importing the web framework. A single error handler in the server translates a ServiceError into the right HTTP response, which keeps the routes free of repeated status-code logic. The named building blocks on the client side, AuthProvider and useAuth in auth.jsx, the apiRequest helper in api.js, and the Login, Inventory, and ItemForm pages, talk to those routes, while requireAuth and requireRole in the server middleware guard them. The credentialsSchema and itemSchema Zod definitions validate input at the boundary.

One design decision is worth explaining directly. In the Module 1 code review I planned to use PostgreSQL as the database. During the build I switched to SQLite. PostgreSQL requires a separate database server to be installed, running, and configured before the application will start, which would make the artifact difficult for a reviewer to actually run. SQLite is an embedded database that ships as part of the project, so the application runs after a single dependency install with no external setup. The relational design, the schema, the constraints, and the parameterized queries are all identical to what they would be with PostgreSQL; only the database engine changed. I judged that a portable, immediately runnable artifact was worth more than matching the original plan exactly, and choosing the right tool for the constraints of the project is itself a software design skill.

Reflection on the process

The hardest part was not writing any single feature but keeping the contract between the frontend and the backend consistent. The React client and the Express server are separate programs, and every endpoint has an agreed shape for its request and its response. The clearest example came from the edit screen. ItemForm.jsx tried to load a single item by calling a GET-by-id endpoint that the backend never actually implemented, so the edit page came up empty with no obvious error. I traced the failure by watching the network request the client made, comparing it against the routes the server really exposed, and confirming that the path the client expected returned nothing. Rather than add an endpoint the rest of the application did not need, I reconciled the client to the real contract: around line 55 of ItemForm.jsx the form now fetches the existing inventory list through apiRequest and selects the matching item on the client by its id. That incident taught me to treat the API contract as a fixed specification and to build both sides against it rather than changing one freely.

Dr. Bolton's Milestone Two feedback shaped the second half of this work, and I want to be specific about how I incorporated each point. She asked me to apply the Single Responsibility Principle and isolate the core business logic from the HTTP routing and the database calls, and that is exactly what the service-layer extraction does: the routes now handle HTTP, the services hold the business rules, and serviceError.js keeps the two from leaking into each other. She asked for explicit encapsulation and a clear sense of what is public versus internal. I addressed that at the module boundary rather than with Java-style access keywords, which JavaScript does not have. Each service exports only its intended public functions, registerUser and loginUser, or the four inventory functions, while the helpers, the prepared SQL statements, and the token-signing routine stay unexported and therefore unreachable from outside the module. I noted that reasoning in the comments so it reads as a deliberate choice. She asked that comments explain the why and not restate the what, so I rewrote the comments to record decisions, such as why bcrypt uses ten salt rounds and why the service layer must not import Express, instead of narrating the obvious. She asked for compliance with an industry style guide, and the project now passes ESLint with no warnings on both the client and the server, with an ESLint configuration committed alongside the code. Finally she asked me to name the specific modules I built and to expand on my troubleshooting steps, which is why this narrative points to authService.js, inventoryService.js, and serviceError.js by name and walks through the API-contract-drift incident in detail.

The security work was also more involved than I expected. Adding authentication is straightforward; making sure every protected route actually checks the token through requireAuth, that role enforcement through requireRole cannot be skipped, and that no validation gap lets bad data through took careful, repetitive checking. The SQLite decision described above was another clear lesson of the milestone: the plan from Module 1 met resistance from a practical constraint, and the right response was to adjust the plan for a defensible reason rather than follow it past the point where it still made sense.
