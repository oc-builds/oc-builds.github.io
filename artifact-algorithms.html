Narrative

Enhancement narrative

The full Milestone Three narrative for the algorithms and data structures enhancement.

Describe the artifact

The artifact is the ABCU Course Advisor, a C++ console application I originally built as Project Two for CS 300, Data Structures and Algorithms, earlier in the SNHU computer science program. The original loads a CSV catalog of computer science courses into a binary search tree, prints the catalog in alphabetical order using an in-order traversal, and looks up a single course and its prerequisites. It is one 397-line file. It works, and the comments are honest about what it cannot do. The BST has no rebalancing. Prerequisites are stored as raw strings with no check that they match real courses. There is no defense against a catalog with circular prerequisites. For Milestone Three I rebuilt the engine underneath the same menu and addressed all three.

Justify the inclusion of the artifact and the improvements made

I picked this artifact because its weaknesses are the exact problems an algorithms enhancement should fix, and because the trade-offs are explainable to a non-technical reader if you do the work. Three things changed under the hood. The unbalanced BST was replaced with a self-balancing AVL tree that tracks node heights and rotates on every insert that pushes a balance factor past plus or minus one (LL, RR, LR, RL). A directed prerequisite graph is built alongside the tree in a two-pass load, because the original CSV references CSCI200 on row two before declaring it on row eight, and a single-pass loader would reject the file. A depth-first search with three-color marking walks the graph at load time and reports any cycle by exact course IDs. The same pass also flags any prerequisite that is referenced but never defined. The menu the user sees is unchanged. The work is all underneath.

The Big-O story matters most for this category. The original BST insertion is O(log n) on average and O(n) in the worst case when input arrives in sorted order. A course catalog exported from a registrar system is a realistic adversarial input, since a human or an export tool will routinely alphabetize by course ID. The 8-row CSV that ships with the project happens to be in a mixed order that avoids the worst case, so I added a second CSV (ABCU_Advising_Program_Input_Sorted.csv) that is sorted by ID and would force the original BST to degenerate. The AVL gives O(log n) on every insert and search regardless of input order, because the rotations keep the balance invariant on every node. Graph construction is O(V + E) where V is courses and E is prerequisite relationships. The DFS cycle detector is also O(V + E), because each vertex is colored at most twice on its way from white to gray to black. The in-order traversal is still O(n) work, but the recursion depth is now O(log n) instead of the O(n) it could become on a degenerate tree, so the call stack is bounded.

One design decision is worth explaining directly because it diverges from a strict reading of the enhancement plan. The Module 1 plan described prerequisite validation as a hard rule that would reject any invalid catalog. During the build I realized that reject is the wrong default. A registrar running this on a catalog with one typo or one cyclic reference still wants to see the list of valid courses so they can find the problem. So the enhanced version warns and continues. The cycle is printed by exact ID. The missing prerequisites are listed by exact ID. The menu still opens. That is a more useful program, and the change is documented in the source and the README so the next reader sees the reasoning.

The code was structured per Dr. Bolton's Single Responsibility guidance from Milestone Two. The AVL tree, the prerequisite graph, the CSV loader, and the menu UI live in separate files, so each one has one job and the tree never needs to care how the graph or the CSV work. Every class hides its internals behind private members and a small public API, and no raw pointers escape any class. The AVL destructor walks the tree post-order and deletes every node, and the classes are marked non-copyable so the owning pointers cannot be double-freed. Every public method and every private helper carries a Big-O note in its header comment, including the four rotation helpers, the graph builder, the DFS visitor, and the in-order traversal. Comments document the why rather than the what, which is the specific thing Dr. Bolton called out in Milestone Two feedback. The AVL header, for example, explains that AVL was chosen because a sorted exporter is a realistic adversarial input, not that the file is an AVL tree.

Course outcomes

In Module 1 I planned for this enhancement to support three course outcomes, primarily Outcome 3 (algorithmic design and trade-offs), and it does. Outcome 3 is met by the AVL versus BST analysis with Big-O, by the choice of three-color DFS over a parent-tracking cycle detector, and by the two-pass loader decision, which is a small but real trade-off that the original plan missed. Outcome 4 (well-founded techniques) is met because AVL trees and three-color DFS are textbook algorithms applied correctly with their published complexity guarantees. Outcome 5 (security mindset) is met through input validation as a first-class concern. The loader treats the CSV as untrusted, the validator catches missing references, and the cycle detector rejects nonsense advising paths that would otherwise propagate. Outcome 2 (communication) is met through this narrative, the README, and the Big-O notes in every header. I have no changes to the outcome-coverage plan. Outcome 1 (collaborative environments) is planned for the Module 5 databases enhancement narrative, where it is the natural fit.

Reflection on the process

The hardest part of this build happened during planning, not coding. My first draft of the justification claimed the original CSV "degenerates" the BST because it loads in sorted order. When I actually opened the file I saw the rows are not in sorted order. They are in a mixed order that produces a perfectly fine branchy tree. The AVL advantage exists, but it exists for adversarial input, not for the shipped catalog. I caught it before writing any code, which is the only reason this narrative does not lean on a false claim. The small lesson there is to look at the actual data before defending a design choice.

A similar thing happened with the CSV loader. A single pass through the file seems obvious at first. You read each row, add the course, add its prerequisite edges. But the original CSV's second row references CSCI200, which is not declared until the eighth row. A single pass would reject the file at row two for a missing prerequisite that actually exists. Splitting the load into a pass that builds the node set first and a pass that resolves edges second was easy to overlook in pseudocode. It became obvious only after walking through a real file in my head.

What I will take from this milestone is the difference between a correct rule and a useful rule. The plan said the loader should reject any invalid catalog. The implementation warns and continues. Reject is easy to defend in a textbook. Warn, show the cycle by exact name, and let the user inspect the data is what an advisor actually needs from this program. That trade-off is, in a small way, what Outcome 3 is really asking about. The work is picking the behavior that fits the real use case, not the one that looks cleanest on paper.
